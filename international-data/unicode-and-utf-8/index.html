Introducing: UTF-32, UTF-16, and UTF-8!

The UTF Crew

Lots of people still use ASCII because it‚Äôs nice and small, and was the default for very many years. 32-bit numbers take up, well, 32 bits, while an 8-bit number can‚Äôt count as high, but only takes up 8 bits.

So: if you use an 8-bit encoding instead of a 32-bit encoding, your files will take up one-quarter as much space. That means a 5MB 8-bit file would be 20MB if it was saved as a 32-bit file.

UTF stands for the easy-to-remember ‚ÄúUniversal Coded Character Set + Transformation Format,‚Äù and (sometimes) tries to solve that problem.

What‚Äôs the difference between Unicode and UTF-xx? Unicode is the system of code points that maps a number to a symbol. These UTF things are the format in which you store those numbers (it‚Äôll hopefully make more sense when you read through).

UTF-32

When we wanted to get an ASCII A, we‚Äôd just use the binary for 65: 100 0001. If we used extended ASCII, it‚Äôs still 65, it just has an extra bit, so the binary number is a little longer: 0100 0001.

If we want a capital B, we‚Äôd need to count to 66, which is the equally-absurd 0000 0000 0000 0000 0000 0000 0100 0010. Just the word ‚ÄúCat‚Äù would look something like:

If you‚Äôre a little worried about all of those seemingly-useless 0s at the beginning of every number, you might use UTF-16 instead.

UTF-8

But the one you‚Äôve probably heard of is UTF-8, which uses only 8 bits. It seems like it wouldn‚Äôt allow you to represent much at all, but you‚Äôd be wrong! Variable encoding to the rescue again.

Remember how ASCII was only 7 bits? Those seven bits of 100 0100 are 67 which is translated into C. It wasn‚Äôt until extended ASCII that they expanded to using an eighth bit.

UTF-8 is perfectly compatible with ASCII values, and it handles them beautifully. As long as the first bit is zero, (the 128ths place) UTF-8 knows you‚Äôre only looking for ASCII values. It‚Äôs like having a 7-bit value with a 1-bit placeholder.

When you‚Äôre saving something more than 7 bits (code points greater than 127) with UTF-8, there‚Äôs a little bit of extra room taken up, i.e. it takes more than 16 bits to store a 16-bit character. That room is 1) the signal that you‚Äôre about to store a bigger number, and 2) the metadata about how big that bigger number is.

Usually this isn‚Äôt a big deal. If you splash an emoji in here or there in what‚Äôs normally un-accented English (standard ASCII), most characters only take up 8 bits.

But! Things get complicated if you‚Äôre using a CJK language (Chinese, Japanese, Korean). Because there‚Äôs no reason to waste your time yelling ‚Äúthis next piece is going to be 16-bits‚Äù over and over again, in those sorts of situations it‚Äôs usually better to use UTF-16. It ends up saving a lot of space that is taken up by those variable-length codes.

TL;DR

Once upon a time, ASCII was invented. It was fine, but caused a lot of trouble.

Then one day Unicode showed up, which currently represents about 120,000 characters but could theoretically represent over a million. Instead of 7 or 8 bits, it uses 32 bits to represent each of those characters. Most of the time you don‚Äôt need to count to the higher numbers, though, so it creates a lot of wasted space.

To combat the size of 32-bit UTF-32 files, UTF-8 cheats and allows you to use abbreviate out some of of the 0s for smaller numbers. UTF-16 does this to a smaller degree, but is probably better for non-Latin languages.
